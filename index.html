Compiler Core Concepts

Foundational concepts every compiler student should know.

🔧 What is a Compiler?

A compiler is a special program that translates source code written in one programming language (the source language) into another language (the target language) — usually machine code or an intermediate form. This translation happens before execution.

Formal Definition: A compiler is a software tool that takes a program as input, performs lexical, syntactic, and semantic analysis, and produces an equivalent lower-level program as output — typically without executing the source program.

The key components of a compiler include a front end (analysis phase) and a back end (synthesis phase).

// Source code input (C) int main() { int x = 10; return x + 5; } // After compilation → Assembly output MOV eax, 10 ADD eax, 5 RET

⚖️ Advantages & Disadvantages

✅ Advantages
❌ Disadvantages
  • Fast execution: Compiled programs run significantly faster since translation is done once before runtime.
  • Optimized code: Compilers can apply sophisticated optimizations like dead code elimination, loop unrolling, and register allocation.
  • Error detection at compile time: Many errors (type mismatches, undeclared variables) are caught before execution.
  • Security: Source code is not distributed; only compiled binary is shipped.
  • Platform-specific output: Can generate highly optimized machine code for a specific processor.
  • Slower development cycle: Every change requires recompilation before testing.
  • Platform dependency: Compiled output is typically platform-specific (x86 vs ARM).
  • Complexity: Compiler construction is significantly more complex than interpreter development.
  • Less flexible: Cannot easily handle highly dynamic code without significant overhead.
  • Larger binary size: Compiled files may be larger than the source for simple programs.

🏗️ Structure of a Compiler

A compiler is broadly divided into two phases:

  • Analysis (Front End): Lexical Analysis → Syntax Analysis → Semantic Analysis → Intermediate Code Generation
  • Synthesis (Back End): Code Optimization → Code Generation

Supporting these phases is the Symbol Table (stores identifiers and their attributes) and the Error Handler (reports and recovers from errors).

💡 The division into front end and back end allows a compiler to be retargetable — you can swap out the back end to generate code for a different CPU without changing the front end.

🌲 Parsers

A parser (syntax analyser) takes the token stream produced by the lexer and checks whether it conforms to the language's grammar. It builds a parse tree or AST and reports syntax errors.

Recursive Descent
Predictive
LR Parsers

Each non-terminal is a function. The function tries to match the input and calls other functions for sub-rules. Supports backtracking.

// Grammar: E → T | T + E function parseE() { parseT(); if (lookahead == '+') { match('+'); parseE(); } } function parseT() { if (lookahead == 'id') match('id'); else if (lookahead == '(') { match('('); parseE(); match(')'); } }

Uses a parse table (built from FIRST/FOLLOW) to select productions without backtracking. Requires LL(1) grammar.

Stack: $ E | Input: id + id $
Consult table[E][id] → E → T E' → push T E'
Continue matching until stack = $

Bottom-up parsers (SLR, LALR, CLR). Read input left-to-right, produce rightmost derivation in reverse. More powerful than LL.

🔁 Passes of a Compiler

A pass is a complete traversal of the source program (or its representation). Each pass reads input, processes it, and writes output for the next pass.

Single-Pass Compiler: All phases complete in one scan (Pascal, older C). Fast, but limited optimization.

Multi-Pass Compiler: Multiple scans allow better analysis and optimization (GCC uses 3–4 passes).

🥾 Bootstrapping

Bootstrapping is the process of writing a compiler for language X in language X itself. The challenge: how do you compile a compiler written in a language that has no compiler yet?

T-Diagram (Tombstone Diagram):
A compiler is described as: Source → Target (written in Impl)
Notation: CS→T,I means: compiles S to T, written in I
// Bootstrap steps for a new language L: Step 1: Write a simple L compiler in assembly (Lasm) → This gives us CL→asm, asm Step 2: Write a better L compiler IN L itself → Source: CL→asm, L Step 3: Compile Step-2 compiler using Step-1 compiler → Now we have CL→asm, asm (full featured) Step 4: Compile Step-2 compiler again using Step-3 → Self-hosting compiler! Discard Step-1 compiler. // Real example: GCC was bootstrapped this way. // Java's javac compiler is written in Java itself.

🌳 Derivation & Parse Tree

A derivation is a sequence of production rule applications starting from the start symbol to produce a terminal string. A parse tree graphically represents that derivation.

Leftmost Derivation
Rightmost Derivation
Parse Tree

Always expand the leftmost non-terminal. Used by LL parsers.

// Grammar: E→E+T|T, T→T*F|F, F→(E)|id // Derive: id + id * id (Leftmost) E ⟹ E + T ← expand E (leftmost) ⟹ T + T ← expand E again ⟹ F + T ← expand T ⟹ id + T ← expand F → id ⟹ id + T * F ← expand T ⟹ id + F * F ← expand T ⟹ id + id * F ← expand F ⟹ id + id * id ✅

Always expand the rightmost non-terminal. Used by LR parsers (reverse rightmost derivation).

// Derive: id + id * id (Rightmost) E ⟹ E + T ← expand T (rightmost) ⟹ E + T * F ← expand F ⟹ E + T * id ← F → id ⟹ E + T * id ← expand T ⟹ E + F * id ⟹ E + id * id ⟹ T + id * id ⟹ F + id * id ⟹ id + id * id ✅

Parse tree for id + id * id — each internal node is a non-terminal, leaves are terminals.

⬆️ Shift-Reduce Parsing

Shift-Reduce is the fundamental mechanism behind all LR parsers. It uses a stack and an input buffer. At each step it either shifts the next input token onto the stack or reduces a handle on top of the stack to a non-terminal.

4 Possible Actions:
Shift — push next input token onto stack
Reduce — replace handle on stack with LHS non-terminal
Accept — input consumed, start symbol on stack
Error — no valid action exists

Example: Parse id * id + id with grammar E→E+E|E*E|id

StepStackInputAction

2$ id* id + id $Reduce E → id
3$ E* id + id $Shift *
4$ E *id + id $Shift id
5$ E * id+ id $Reduce E → id
6$ E * E+ id $Reduce E → E*E
7$ E+ id $Shift +
8$ E +id $Shift id
9$ E + id$Reduce E → id
10$ E + E$Reduce E → E+E

🧮 Operator Precedence Parsing

A simplified shift-reduce method for expressions. It uses an operator precedence table to decide between shift and reduce using three relations: < (yields precedence), = (equal precedence), > (takes precedence).

Precedence Rules:
a <· b means b has higher precedence than a (shift b first)
a ·= b means a and b have equal precedence (associativity decides)
a ·> b means a has higher precedence (reduce a first)
id+*$
id—·>·>·>

*<··>·>·>
$<·<·<·acc

Parse id + id * id: Shift id, reduce E→id; shift +; shift id; reduce E→id; since * > + in precedence, shift *; shift id; reduce E→id; reduce E→E*E; reduce E→E+E. ✅

🔨 YACC — Yet Another Compiler Compiler

YACC (and its GNU equivalent Bison) is a parser-generator tool. You write a grammar in a specification file; YACC generates a C-based LALR(1) parser automatically.

/* calc.y — YACC specification for a calculator */ %{ #include <stdio.h> int yylex(); %} %token NUM /* token declaration */ %left '+' '-' /* left-associative, lower precedence */ %left '*' '/' /* left-associative, higher precedence */ %% expr : expr '+' expr { $$ = $1 + $3; } | expr '-' expr { $$ = $1 - $3; } | expr '*' expr { $$ = $1 * $3; } | expr '/' expr { $$ = $1 / $3; } | NUM { $$ = $1; } | '(' expr ')' { $$ = $2; } ; %% /* yacc generates: yyparse() function */ /* Compile: yacc -d calc.y → y.tab.c + y.tab.h */
YACC Workflow:
1. Write .y file (grammar + semantic actions)
2. Run yacc -d calc.y → generates y.tab.c and y.tab.h
3. Write .l file for Lex (lexer)
4. Compile: cc lex.yy.c y.tab.c -ly -ll

SDT attaches semantic rules (actions) to grammar productions. When the parser reduces using a production, the associated semantic action executes. There are two forms:

SDD (Attributes)
SDT (Actions)

A Syntax-Directed Definition (SDD) associates attributes with grammar symbols. Synthesized attributes flow up the tree; inherited attributes flow down.

🔄 Type Conversion

Type conversion changes a value from one data type to another. Compilers handle two kinds:

Implicit (Coercion)
Explicit (Cast)
TAC with Types

Implicit conversion (coercion) is done automatically by the compiler when operands of mixed types appear in an expression. A widening conversion (e.g. int → float) is always safe.

Type Widening Hierarchy (safe — no data loss)
char→ short→ int→ long→ float→ double (wider = more precision)
1
Source Code — mixed int + float expression
int i = 5; float f = i + 2.5f; // i is int, 2.5f is float
2
Compiler detects type mismatch → inserts implicit cast
int is narrower than float, so compiler promotes i to float automatically.
3
Generated TAC (Three-Address Code)
t1 = int_to_float(i) // promote i → float t2 = t1 + 2.5 // float addition f = t2 // assign result

Explicit cast is programmer-initiated. Narrowing conversions (e.g. double → int) may lose data and require explicit cast.

1
Source Code — explicit cast (programmer writes the cast)
double d = 9.99; int i = (int) d; // explicit cast: i = 9 (truncated!)
2
Generated TAC — compiler inserts truncation instruction
t1 = double_to_int(d) // truncate decimal part i = t1 // i = 9 (0.99 is lost!)
3
SDT semantic action that generates the cast
E → E₁ op E₂ { if E₁.type == "int" && E₂.type == "float" E₁ = gen("int_to_float", E₁) // widen E₁ E.type = "float" // result is float }
Compiler Rule: Result type = the wider of the two operands.
Hierarchy: char < short < int < long < float < double
1
Source — int + double mixed expression
int a = 3; double b = 4.5; double c = a + b; // a must be promoted
2
Compiler promotes a from int → double

Parsing Techniques

Understanding how parsers analyze syntax and build parse trees from token streams.

📌 What is Parsing?

Parsing (Syntax Analysis) is the second phase of a compiler. It takes the token stream produced by the lexer and checks whether the tokens form a grammatically valid program according to the language's formal grammar.

A parser produces a Parse Tree (or Abstract Syntax Tree) that represents the hierarchical structure of the source program. This tree guides subsequent phases.

⬇️ Top-Down Parsing

Top-down parsers build the parse tree from the root (start symbol) down to the leaves (terminals), attempting to derive the input string.

  • Recursive Descent Parser: A hand-written top-down parser where each non-terminal corresponds to a function. Simple and flexible, supports backtracking.
  • Predictive Parser (LL(1)): Non-backtracking top-down parser that uses a lookahead of 1 token and a parse table to make decisions deterministically.
  • Backtracking vs Non-backtracking: Backtracking parsers try alternatives on failure; predictive parsers select the unique correct production from the parse table.
// Recursive Descent pseudo-code for E → T E' function parseE() { parseT(); parseEPrime(); } function parseEPrime() { if (lookahead == '+') { match('+'); parseT(); parseEPrime(); } // else: ε production }

⬆️ Bottom-Up Parsing

Bottom-up parsers build the parse tree from the leaves up to the root by repeatedly reducing substrings to non-terminals.

  • Shift-Reduce Parsing: Uses a stack and input buffer. Either shifts the next token onto the stack or reduces a handle on the stack to a non-terminal.
  • LR Parsers: More powerful than LL parsers. Types include LR(0), SLR(1), LALR(1) (used in Yacc/Bison), and CLR(1). They read input Left-to-right and produce a Rightmost derivation in reverse.

🔄 Left Recursion & Left Factoring

Left recursion occurs when a non-terminal can derive a string beginning with itself, e.g. A → Aα | β. This causes infinite loops in top-down parsers and must be eliminated.

Elimination formula:
A → Aα | β  ⟹  A → β A'  and  A' → α A' | ε

Example with expression grammar:

// Before (left recursive) E → E + T | T // After elimination E → T E' E' → + T E' | ε

Left Factoring is applied when multiple productions for the same non-terminal share a common prefix, which also prevents predictive parsing:

// Before left factoring A → αβ | αγ // After left factoring A → α A' A' → β | γ

Automata Theory

Finite automata underpin lexical analysis. Learn DFA, NFA, and how to convert between them.

🔵 DFA — Deterministic Finite Automaton

For each state and input symbol, exactly one transition exists. Used directly in lexical analyzers.

StateInput: aInput: b
→ q0q1q0
* q1q1q0

🟣 NFA — Non-deterministic Finite Automaton

Multiple transitions from one state on same symbol. ε-transitions (free moves) allowed. Recognized by lexers via subset construction.

Stateabε
→ q0{q0, q1}{q0}{q1}
q1∅{q2}∅
* q2∅∅∅

🔄 NFA to DFA Conversion (Subset Construction)

The subset construction algorithm converts an NFA to an equivalent DFA by treating sets of NFA states as single DFA states.

Algorithm:
1. Start with ε-closure(q0) as initial DFA state
2. For each DFA state S and symbol a: compute ε-closure(move(S, a))
3. If the result is new, add it as a DFA state
4. Repeat until no new states are created
5. DFA states containing NFA accepting states are accepting states

Example: NFA above → DFA states: {q0}, {q0,q1}, {q0,q2}, {q0,q1,q2}

A{q0,q1}BCNo
B{q0,q1}BDNo
C{q0,q2}BCYes (q2∈C)
D{q0,q1,q2}BDYes (q2∈D)

Grammar & Context-Free Grammars

Formal grammars define the syntax of programming languages. CFGs are the backbone of parsing.

📐 Formal Grammar (Type-2 in Chomsky Hierarchy)

A grammar G = (V, T, P, S) where:

  • V: Set of variables (non-terminals)
  • T: Set of terminals (actual symbols)
  • P: Set of production rules of the form A → α
  • S: Start symbol (S ∈ V)
// Example CFG for arithmetic expressions E → E + T | T T → T * F | F F → ( E ) | id | num

🌿 Derivations

Leftmost
Rightmost
Ambiguity

A leftmost derivation always expands the leftmost non-terminal first.

E ⟹ E + T (E+T) ⟹ T + T (expand leftmost E) ⟹ F + T (expand leftmost T) ⟹ id + T (expand F) ⟹ id + T * F (expand T) ⟹ id + F * F (expand T) ⟹ id + id * F (expand F) ⟹ id + id * id (expand F)

A rightmost derivation always expands the rightmost non-terminal first (used in LR parsing).

E ⟹ E + T ⟹ E + T * F ⟹ E + T * id ⟹ E + F * id ⟹ E + id * id ⟹ T + id * id ⟹ F + id * id ⟹ id + id * id

⚠️ Ambiguous Grammars

A grammar is ambiguous if a string has more than one parse tree (or more than one leftmost/rightmost derivation). For example, the grammar E → E + E | E * E | id is ambiguous — id + id * id has two parse trees.

❗ Ambiguous grammars are problematic for compilers because the meaning of the program is undefined. Ambiguity is resolved by rewriting the grammar to enforce operator precedence and associativity.

🔢 Chomsky Normal Form (CNF)

A CFG is in CNF if every production is of the form: A → BC or A → a. CNF is useful in the CYK parsing algorithm and for proving properties about CFLs.

🏛️ The Chomsky Hierarchy

Proposed by Noam Chomsky in 1956, the Chomsky Hierarchy classifies all formal grammars into four nested types based on the complexity of their production rules. Each type defines a class of languages and corresponds to a specific computational model.

Key Insight: Each type is a strict superset of the type below it — every regular language is also context-free, every context-free language is also context-sensitive, and so on. The hierarchy defines the expressive power of grammars.

Type 0 — Unrestricted Grammars

The most powerful type. Productions can be of the form α → β where α and β are any strings of terminals and non-terminals (α must contain at least one non-terminal). These grammars generate recursively enumerable languages — the same languages recognizable by a Turing Machine.

Production Rule:   α → β   where α ∈ (V∪T)* V (V∪T)* and β ∈ (V∪T)*
Automaton: Turing Machine (TM)
Language class: Recursively Enumerable Languages (RE)
// Type 0 example: string duplication language {ww | w ∈ {a,b}*} // Productions allow arbitrary rewrites — can simulate any computation S → aSBb | ε bB → Bb aB → ab // Non-terminals can appear anywhere on both sides of → // This power makes them impossible to parse efficiently
  • Closure properties: Closed under union, concatenation, Kleene star, complement
  • Decidability: Membership is semi-decidable (halting problem applies)
  • Use in compilers: Not directly used — too powerful and undecidable
1

Type 1 — Context-Sensitive Grammars (CSG)

Productions are of the form αAβ → αγβ — a non-terminal A can only be replaced by γ in the context of surrounding strings α and β. The key constraint is that |αγβ| ≥ |αAβ| — productions never shrink the string (except S → ε if S doesn't appear on any RHS).

Production Rule:   αAβ → αγβ   where A ∈ V, α,β,γ ∈ (V∪T)*, γ ≠ ε
Constraint: Length of RHS ≥ Length of LHS (non-shrinking)
Automaton: Linear Bounded Automaton (LBA) — a TM with tape bounded by input length
Language class: Context-Sensitive Languages (CSL)
// Classic CSL: L = {aⁿbⁿcⁿ | n ≥ 1} — cannot be expressed as a CFG S → aSBC | aBC CB → BC // context-sensitive: B replaced next to C aB → ab bB → bb bC → bc cC → cc // Note: context on BOTH sides of A constrains what replaces it
  • Example languages: aⁿbⁿcⁿ, {ww | w∈Σ*}, {aⁿ² | n≥1}
  • Decidability: Membership is decidable (LBA recognizes them)
  • Use in compilers: Some semantic constraints (e.g. variable scoping) are context-sensitive — but handled via symbol tables rather than CSG
2

Type 2 — Context-Free Grammars (CFG) ← Used in Compilers

The backbone of compiler design. Productions have the form A → γ — a single non-terminal on the left replaced by any string. The non-terminal is replaced regardless of its surrounding context. Most programming language syntax is described by CFGs.

Production Rule:   A → γ   where A ∈ V, γ ∈ (V∪T)*
Automaton: Pushdown Automaton (PDA) — a finite automaton with a stack
Language class: Context-Free Languages (CFL)
Parser types: LL(k), LR(k), LALR, SLR, Earley, CYK
// Classic CFG: L = {aⁿbⁿ | n ≥ 1} — CFG but not regular S → aSb | ab // Programming language CFG for expressions E → E + T | T T → T * F | F F → ( E ) | id // Single non-terminal A on LHS, any string on RHS // No context needed — A is replaced regardless of surroundings
  • Example languages: aⁿbⁿ, balanced parentheses, arithmetic expressions, if-else structures
  • Decidability: Membership decidable in O(n³) via CYK algorithm
  • In compilers: Used for syntax analysis (parsing) — the grammar for C, Java, Python is a CFG
  • Limitations: Cannot express cross-serial dependencies like aⁿbⁿcⁿ (need CSG)
3

Type 3 — Regular Grammars ← Used in Lexical Analysis

The most restrictive type. Productions are either right-linear (A → aB or A → a) or left-linear (A → Ba or A → a) — but not both in the same grammar. These grammars generate regular languages, which are also described by regular expressions and recognized by DFA/NFA.

Right-Linear Rule:   A → aB or A → a   (a ∈ T, B ∈ V)
Left-Linear Rule:   A → Ba or A → a   (cannot mix with right-linear)
Automaton: Deterministic / Non-deterministic Finite Automaton (DFA/NFA)
Language class: Regular Languages
// Regular grammar for L = {aⁿb | n ≥ 0} — right-linear S → aS | b // matches a*b // Regular grammar for identifiers: letter (letter | digit)* S → lS | dS | l // l = letter, d = digit // Equivalent regular expression: [a-zA-Z][a-zA-Z0-9]* // Equivalent DFA: states track whether we have seen ≥1 letter // Note: aⁿbⁿ is NOT regular — requires unbounded counting (needs a stack)
  • Example languages: All finite languages, aⁿ, (a|b)*, identifiers, keywords, integer literals
  • Decidability: Membership decidable in O(n) — just simulate the DFA
  • In compilers: Used for lexical analysis (tokenization) — all token patterns are regular
  • Pumping Lemma: If L is regular, then strings long enough can be "pumped" — used to prove languages are NOT regular

📋 Rules for Computing FIRST(α)

FIRST(α) is the set of terminals that can appear as the first symbol in strings derived from α.

Rule 1 — Terminal
If α = a (a terminal), then FIRST(α) = {a}
Rule 2 — ε Production
If α = ε, then FIRST(α) = {ε}
Rule 3 — Non-terminal A with A → ε
Add ε to FIRST(A)
Rule 4 — Non-terminal A with A → X1X2...Xk
Add FIRST(X1) − {ε}. If ε ∈ FIRST(X1), add FIRST(X2) − {ε}, etc. If all Xi can derive ε, add ε.

📋 Rules for Computing FOLLOW(A)

FOLLOW(A) is the set of terminals that can appear immediately after A in any sentential form.

Rule 1 — Start Symbol
$ ∈ FOLLOW(S) ($ = end-of-input marker)
Rule 2 — A → αBβ
Add FIRST(β) − {ε} to FOLLOW(B)
Rule 3 — A → αB or A → αBβ where ε ∈ FIRST(β)
Add FOLLOW(A) to FOLLOW(B)
